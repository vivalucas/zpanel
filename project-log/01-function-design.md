## 登录与用户

### 功能列表

- [x] 用户登录 / 登出。
- [x] 根据 token 获取当前用户认证信息。
- [x] 管理员维护用户。
- [x] 用户修改基础资料。
- [x] fork 后默认账号从上游地址改为 `admin@zpanel.local`。
- [x] 登录图形验证码，可通过系统配置开关启用。
- [x] 多账号快速切换，前端本地保存已登录账号 token。
- [ ] 继续梳理默认密码策略和重置密码能力。

### 业务逻辑

```text
用户提交登录信息 → 后端校验用户名和密码 → 写入 / 返回 token → 前端后续请求携带 token
```

开启登录验证码后：

```text
前端生成 captchaId → 后端按 captchaId 生成验证码图片并缓存答案 → 登录时提交 captchaId + vcode → 后端一次性校验
```

### 边界情况

- 未登录请求需要被登录拦截器拒绝。
- 停用或未激活账号不能正常登录。
- 默认管理员账号来自初始化逻辑，当前为 `admin@zpanel.local / 12345678`；默认密码仍需后续改进。
- 多账号快速切换依赖浏览器本地存储，不是服务端会话列表；清理浏览器数据后需要重新登录。

## 面板导航

### 功能列表

- [x] 导航项目新增、编辑、删除、批量新增。
- [x] 导航项目排序保存。
- [x] 按分组获取导航项目。
- [x] 导航项目分组新增、编辑、删除、排序。
- [x] 公开模式下读取允许公开展示的项目。

### 业务逻辑

```text
用户配置导航项目 → 后端按用户和分组持久化 → 前端按分组渲染 → 用户可按网络模式打开不同地址
```

### 边界情况

- 公开模式下不能暴露需要登录的管理接口。
- 删除分组时需要关注组内导航项目如何处理，当前以现有代码行为为准，后续需要补文档。
- 批量导入仅支持 ZPanel 的 `.zpanel.json`。

## 文件与图标

### 功能列表

- [x] 上传图片。
- [x] 设置上传图片为壁纸。
- [x] 公共图库视图。
- [x] 获取站点 favicon。
- [x] 图标和样式配置导入导出。

### 业务逻辑

```text
用户上传 / 选择图标 → 文件保存到 data/uploads → 数据库存储文件对象 → 前端渲染资源
```

目标业务逻辑：

```text
用户上传资源 → 创建 files 记录和 object_key → 按 owner/purpose 保存物理文件
  → 业务实体保存 file_id → file_references 记录引用关系 → 前端通过资源 API 渲染
```

### 边界情况

- `source_path` 是旧配置名，新配置模板已移除，上传根目录由 `storage.uploads_path` 控制。
- 图标导入导出格式按 ZPanel 新格式维护，不兼容旧 Sun-Panel 导入文件。
- 公共图库目前是全局图片列表视图；删除仍调用当前用户文件删除逻辑，不能删除他人文件。
- 当前按日期目录保存上传文件，不利于替换、删除和人工排查；后续按 `02-database-design.md` 改为 object key + owner/purpose 目录。
- 删除文件前需要检查引用关系；仍被导航图标、壁纸、头像、站点图标、登录背景使用的资源不能直接物理删除。

## Docker 管理

### 功能列表

- [x] 管理员查看容器列表。
- [x] 管理员查看 Docker stats 快照。
- [x] 管理员启动、停止、重启容器。
- [x] 管理员查看容器日志。

### 业务逻辑

```text
管理员打开 Docker 管理 → 前端调用后端 Docker API → 后端执行 Docker CLI → 返回容器状态 / 日志
```

### 边界情况

- 部署环境必须能执行 `docker` 命令。
- 如果 ZPanel 在容器中运行，需要挂载 `/var/run/docker.sock` 才能管理宿主机容器。
- Docker socket 权限很高，该功能必须只允许管理员访问。

## 本地化

### 功能列表

- [x] README 支持与 DashCat 对齐的 11 种语言入口。
- [x] 产品侧支持简体中文、English、日本語、한국어、Deutsch、Français、Español、Português do Brasil、Italiano、繁體中文、Русский。
- [x] 语言选择写入用户配置，默认语言可跟随浏览器语言。
- [x] locale key 已对齐，切换语言时不应因为缺少 key 导致界面异常。
- [x] 中英文产品文案已完成第一轮优化。
- [ ] 日语、韩语、德语、法语、西班牙语、巴西葡萄牙语、意大利语、俄语的真实翻译版。
- [ ] 繁体中文的繁体化、地区用语和标点习惯校对。

### 业务逻辑

```text
用户选择语言 → 保存到用户配置 → 前端 i18n 切换 locale → 各组件从 locale key 读取文案
```

### 边界情况

- README 翻译和产品 UI 翻译是两类工作；README 有对应语言文件，不代表产品 UI 已完成该语言的真实翻译。
- 当前非中英产品 locale 先以英文基线保证 key 完整；繁体中文先以简体中文基线保证可运行。
- 真正翻译版需要逐语言处理术语、长度、按钮截断和界面实测，不能只做机器替换。
