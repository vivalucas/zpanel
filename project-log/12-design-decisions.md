## 决策 4：以独立仓库形态发布 ZPanel 1.0.0

**日期**：2026-05-21

**状态**：已采用

**替代关系**：补充“ZPanel 作为独立维护 fork，而非上游原名镜像”

**类型**：产品 / 仓库治理 / 发布

**背景**：

用户在另一台电脑上删除原 GitHub fork 仓库，并重新创建同名仓库后推送。目标是让 ZPanel 对外呈现为独立开源项目，而不是 GitHub 页面上的 fork 项目。同时仍需在 README / LICENSE 中保留 Sun-Panel 的 MIT 来源说明。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 保留 GitHub fork | 保留 fork 标识和完整历史 | 来源链路直观 | 不符合独立项目外观目标 |
| B. 新仓库但保留全部历史 | 重新创建仓库并推送完整历史 | 历史完整 | 新项目噪声多，仍带较多清理过程 |
| **C. 新仓库 + 精简历史** | 重新创建同名仓库，以初始化和 1.0.0 发布提交为基线 | 对外简洁，维护边界清晰 | 需要在文档中补足背景和历史决策 |

**最终选择**：

选择 C：ZPanel 以重新创建的同名独立仓库发布，公开历史精简为初始化和 1.0.0 发布基线；README / LICENSE 保留上游 MIT 来源和归属。

**依据**：

- 用户明确希望去掉 GitHub fork 标识、Sync fork 和旧历史。
- ZPanel 的产品方向已进入独立维护阶段。
- 上游来源说明保留在 README / LICENSE 中，不依赖 GitHub fork 标识表达。

**实现原则**：

- 对外文档减少对原项目、PRO、付费对比的过度表述。
- 保留必要上游归属，避免版权和用户认知混淆。
- project-log 作为本地知识库记录更详细的历史背景，但整体不提交到 Git。
- 1.0.0 作为独立仓库的第一个公开基线版本。

**已知不足**：

- 因为 project-log 不在 GitHub 上，跨电脑开发时需要额外同步或重建。
- 精简 Git 历史后，早期清理过程主要依赖 project-log 和本地旧副本追溯。
- 当前仍需补齐真实测试覆盖和 Docker 管理实机验证。

**后续触发条件**：

- 每次 release 前检查 README、CHANGELOG、版本号、Docker 镜像 tag 和 GitHub release 是否一致。
- 如果未来需要公开更完整迁移历史，可在文档中补“历史背景”而不是恢复 fork 标识。

## 决策 3：README 与产品语言集合保持一致

**日期**：2026-05-20

**状态**：已采用

**替代关系**：无

**类型**：产品 / 前端 / 文档

**背景**：

用户希望 ZPanel 像 DashCat 一样提供多语言 README，并要求产品本身也支持同样的语言。这个需求包含两层：一是语言入口和产品 i18n 框架，二是每种语言的真实翻译质量。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 继续只支持中英文 | 不增加新的 README 和产品语言 | 维护成本低 | 不符合项目多语言目标 |
| B. 只做 README 多语言 | 产品 UI 仍只保留中英文 | 文档层面容易完成 | 产品体验和文档承诺不一致 |
| **C. 语言集合统一，真实翻译分阶段完成** | README 和产品都支持 11 种语言，先完成框架和 key 对齐，后续逐语言翻译 | 架构清晰，便于持续补齐 | 第一阶段非中英语言还不是最终体验 |

**最终选择**：

选择 C：README 和产品语言集合保持一致，均支持简体中文、English、日本語、한국어、Deutsch、Français、Español、Português do Brasil、Italiano、繁體中文、Русский。当前先完成语言框架、locale key 对齐、中英文优化和文档说明；非中英真实翻译版后续逐个补齐。

**依据**：

- 用户明确要求 README 和产品本身都支持这些语言。
- 先把 locale key 对齐，可减少后续翻译工作中的技术噪声。
- 翻译质量需要语境、术语、UI 长度和地区用语校对，不适合一次性草率宣布完成。

**实现原则**：

- README 多语言和产品 UI 多语言都要有清晰入口。
- 产品 locale 文件必须 key 完整，不能因为切换语言导致运行时缺失文案。
- 中英文作为第一批真实可用文案维护。
- 其他语言在真实翻译完成前只作为基线文案，不对外宣称已经完成母语级翻译。

**已知不足**：

- 日语、韩语、德语、法语、西班牙语、巴西葡萄牙语、意大利语、俄语目前使用英文基线文案。
- 繁体中文目前使用简体中文基线文案。
- 真正翻译版仍需逐语言完成术语表、母语级翻译、界面长度检查和实际交互校验。

**后续触发条件**：

- 发布面向多语言用户的正式版本前，至少应完成目标语言的真实翻译或在发布说明中清楚标注翻译状态。
- 新增用户可见文案时，必须同步更新所有 locale key。
- 每完成一种真实翻译，需要进行桌面和移动视口的 UI 文案溢出检查。

**相关文件**：

- `README.md`
- `README*.md`
- `src/locales/`
- `src/locales/index.ts`
- `src/store/modules/app/helper.ts`
- `src/utils/defaultData/index.ts`
- `10-planning-log.md` 中的 ADR-004
- `06-dev-log.md` 中的 2026-05-20（多语言 README 与产品本地化框架）

**后续补充**：

- 真实翻译版完成后，在本条决策下追加各语言完成状态。

## 决策 2：公开 PRO 功能直接进入 ZPanel 开源版

**日期**：2026-05-20

**状态**：已采用

**替代关系**：无

**类型**：产品 / 安全 / 架构

**背景**：

上游曾以标准版 / PRO 版区分功能。用户提供了公开功能对比表，并要求这些功能在 ZPanel 中直接实现。ZPanel 的定位是全新维护的开源 fork，不准备在当前阶段设计付费授权系统。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 维持功能限制 | 保留标准版 / PRO 版差异 | 接近上游商业模型 | 与 ZPanel 当前开源目标冲突 |
| B. 等待长期插件系统 | 把 PRO 功能推迟到未来 V2 / 插件化后实现 | 架构更从容 | 短期不能解决用户需求 |
| **C. 直接开源化实现** | 按公开描述重新实现并开放给所有 ZPanel 用户 | 用户价值明确，维护边界清晰 | 需要补安全提示和后端验证 |

**最终选择**：

选择 C：公开 PRO 功能表中的面板能力直接进入 ZPanel 开源版，不引入付费限制，不复制闭源代码，也不实现上游商业商城 / 授权后台。

**依据**：

- 用户明确要求“在我们这个版本里面实现它所有的这些功能”。
- 功能表描述的是能力，不是闭源代码本身。
- 当前代码已有用户管理、文件管理、样式配置、导入导出等基础，适合第一版增量实现。
- ZPanel 当前目标是独立维护和自托管可用性，不是商业授权平台。

**实现原则**：

- 配置类能力优先复用 `system_setting` 和既有用户配置。
- 多账号快速切换先做浏览器本地能力，不新增服务端会话管理表。
- Docker 管理只对管理员开放，并明确部署环境权限要求。
- 备份迁移只维护 `.zpanel.json`，不恢复旧 `.sunpanel.json` 兼容。

**已知不足**：

- Go 后端尚未在当前环境完成编译验证。
- Docker 管理未在真实 Docker 环境完成操作验证。
- 自定义 JS / CSS 缺少一键恢复或安全模式。
- 登录验证码请求中当前复用了 `email` 字段传 `captchaId`，后续可重命名以减少歧义。

**后续触发条件**：

- 发布 release 前必须完成 Go 编译、后端启动和 Docker 管理实机验证。
- 如果未来做插件 / 应用市场，需要重新评估 Docker 管理和公共图库是否应插件化。
- 如果用户反馈自定义 JS / CSS 误配置导致无法进入系统，应优先实现安全模式或配置重置入口。

**相关文件**：

- `10-planning-log.md` 中的 ADR-003
- `06-dev-log.md` 中的 2026-05-20（PRO 功能开源化第一版）
- `service/api/api_v1/system/docker.go`
- `src/components/apps/DockerManager/index.vue`
- `src/components/apps/Style/index.vue`

**后续补充**：

## 决策 1：ZPanel 作为独立维护 fork，而非上游原名镜像

**日期**：2026-05-20

**状态**：已采用

**替代关系**：无

**类型**：产品 / 架构 / 部署 / 其他

**背景**：

项目来源于 Sun-Panel。用户长期使用原项目，但上游开源版本长期缺少持续维护，后续 PRO / 付费相关规划也没有按用户预期持续公开更新。为了继续使用和演进该导航面板，需要基于 MIT 协议建立可独立维护的 fork。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 原名镜像 | 保持 Sun-Panel 名称，只在自己仓库维护 | 改动少 | 归属混淆，用户难以区分上游和 fork |
| B. 全面重写 | 以 Sun-Panel 为参考重新实现 | 自由度高 | 成本高，短期无法保证现有功能可用 |
| **C. 独立 fork** | 使用 ZPanel 名称，保留上游归属，逐步维护 | 归属清晰，迁移风险可控 | 初期会有命名残留和兼容问题要清理 |

**最终选择**：

选择 C：ZPanel 作为独立维护 fork。项目名、包名、二进制名、Docker 服务名和主要用户可见文案使用 ZPanel；README、LICENSE 和关于页保留上游 Sun-Panel 归属。

**依据**：

- MIT 协议允许 fork 和修改，但需要保留原版权与许可证。
- 独立命名能降低用户误解，也便于后续发布 Docker 镜像、release 和文档。
- 初始化阶段保留原有业务逻辑，优先减少功能回归。

**实现原则**：

- 用户可见的当前项目入口使用 ZPanel。
- 上游归属不删除、不隐藏。
- 不保留旧 `.sunpanel.json` 导入兼容，ZPanel 只维护自己的 `.zpanel.json`。
- 不把上游闭源 / PRO 商业实现当作当前 fork 已拥有能力来宣传；公开功能表中的能力可在 ZPanel 中重新实现。

**已知不足**：

- 短期内仍可能存在少量上游历史图片或截图资源，需要后续按新品牌替换。
- README 已简化，后续还需要补充更完整的安装和开发说明。
- Go 后端尚未在当前环境完成编译验证。

**后续触发条件**：

- 发布第一个 ZPanel release 前，重新检查品牌、链接、LICENSE、关于页和 Docker 镜像命名。
- 如果未来用户明确提出迁移需求，再单独设计迁移工具，而不是混入常规导入流程。
- 如果未来和上游重新同步代码，需要复查命名冲突和许可证说明。

## 决策 5：管理员级安全能力的边界认定

**日期**：2026-05-22

**状态**：已采用

**替代关系**：无

**类型**：安全 / 产品

**背景**：

第二轮代码评审中，多个安全相关的能力需要判定是 bug 还是设计意图：customJs/CSS 注入、favicon 获取（潜在 SSRF）、默认密码、静态文件无鉴权、Token 明文存储。

**讨论过程**：

项目支持公网和内网两种部署场景。部分用户直接部署在公网服务器上通过域名访问，部分用户部署在家庭内网的 NAS 上。需要结合两种场景判断每个安全项的性质。

**最终判定**：

| 能力 | 判定 | 理由 |
|------|------|------|
| customJs/CSS 直接注入 DOM | 设计意图 | README 明确列出"在线编辑自定义 CSS 和 JavaScript"为功能。管理员自定义能力，文档已提示风险。 |
| GetSiteFavicon 获取内网地址 | 设计意图 | favicon 获取是导航面板核心功能。NAS 部署场景下获取内网服务 favicon 是正常用例。仅登录后的管理员可触发。 |
| 默认密码 12345678 | 设计意图 | 开源项目，源码公开，密码保密无意义。README 已提示"首次登录后请立即修改默认密码"。 |
| /uploads 静态文件无鉴权 | 设计意图 | 自托管面板，上传的图标和壁纸需要被浏览器直接访问。 |
| Token 明文存 localStorage | 标准做法 | SPA 项目标准方案，安全重点应放在防 XSS 而非加密存储。 |

**依据**：

- 项目 README 已将 customJs/CSS、Docker socket、公开访问列为"高权限能力"并明确提示风险。
- 这些能力的设计意图是给管理员使用，不是暴露给任意用户。
- 自托管面板的安全模型与 SaaS 不同，管理员即部署者。

**已知不足**：

- customJs 注入无 CSP 防御纵深。
- GetSiteFavicon 无内网 IP 过滤，公网部署时理论上可被用于探测服务器所在内网。
- 这些不足在当前阶段可接受，后续如需加强可单独补充。

**后续触发条件**：

- 如果项目未来支持多租户或非管理员用户可触发这些能力，需要重新评估。
- 如果用户反馈公网部署时被扫描内网，应优先给 GetSiteFavicon 加 IP 白名单或禁用选项。

**⚠ 注意**：以上判定主要基于 AI 对项目场景的理解，可能不完全准确。如有不同意见请修正。

**相关文件**：

- `src/App.vue`（customJs 注入）
- `service/api/api_v1/panel/itemIcon.go`（GetSiteFavicon）
- `service/initialize/database/connect.go`（默认密码）
- `service/router/router.go`（静态文件）
- `src/store/modules/auth/helper.ts`（Token 存储）

## 决策 6：全新项目无需向后兼容

**日期**：2026-05-22

**状态**：已采用

**替代关系**：无

**类型**：架构 / 安全

**背景**：

第一轮评审中，密码加密采用了 bcrypt + MD5 回退的兼容策略，以支持旧用户迁移。经确认，ZPanel 是全新项目，没有从 Sun-Panel 迁移过来的老用户。

**最终选择**：

移除 MD5 密码回退逻辑。bcrypt 失败时直接报错，不再降级。`VerifyPassword` 中的 MD5 验证分支也一并移除。

**依据**：

- 没有旧用户需要迁移。
- MD5 对密码哈希来说极弱，保留回退逻辑增加攻击面而无收益。
- 全新项目可以干净地使用最佳实践，不需要历史包袱。

**⚠ 注意**：此决策基于"项目没有老用户"的前提。如果未来发现有旧数据导入场景，需要重新评估。

**相关文件**：

- `service/lib/cmn/base.go`（PasswordEncryption、VerifyPassword）
