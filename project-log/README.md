## 每次开工前必读 / 必做流程

每个新成员、新 AI 对话、新开发轮次开始前，先按以下流程同步上下文：

1. 读取 `05-current-status.md`，确认当前阶段、待办、阻塞和下一步。
2. 读取与任务相关的设计文档，例如功能改动读 `01`，数据库改动读 `02`，API 改动读 `03`，架构改动读 `04` 和 `10`。
3. 如果当前设计原因不明，或准备推翻已有产品/交互/架构取舍，先查 `12-design-decisions.md`。
4. 改代码前先说明计划、影响范围和可能需要同步更新的 project-log 文件。
5. 涉及较大改动时，先在 `10-planning-log.md` 记录方案并确认，再动代码。
6. 改完后更新 `05-current-status.md` 和 `06-dev-log.md`。
7. 涉及功能、数据库、API、架构、部署、环境变量或外部服务时，同步更新对应编号文档。
8. 如果本轮形成了长期设计取舍，或放弃了未来可能再次被提出的方案，同步更新 `12-design-decisions.md`。
9. 无法验证、发现文档与代码不一致、或遇到不确定决策时，必须明确写出，不要默认跳过。

## 文档哲学：优先留痕，允许清理

- **优先留痕** — 决策、废弃方案、风险和踩坑记录要保留上下文
- **允许清理** — 明显错误、重复、已经会误导后续开发的内容，可以直接改正或删除
- **记录「改动 + 原因」** — 不只写结果，要写为什么这么做
- **保留被否定的方案** — 「为什么不用 X」和「为什么用 Y」同样有价值
- **不适用也要标注** — 某节暂时不用时，写明「不适用」和原因，不直接删除结构

修改已有内容时按影响选择方式：

- 长期决策或重要历史：保留旧方案并注明废弃原因。
- 普通状态、版本号、待办、已完成项：直接更新为当前事实。
- 明显错误或会误导的描述：直接删除或改正，并在变更记录中说明。

```markdown
<!-- 旧方案（废弃于 2026-05-07，原因：性能不足） -->
~~使用 SQLite 作为主数据库~~

<!-- 新方案 -->
使用 PostgreSQL，详见 10-planning-log.md
```

### 决策流程

较大改动（新功能、架构调整、技术选型变更）前，先在 `10-planning-log.md` 中写清楚方案和决策依据，确认后再动代码。每条决策记录包含：决策编号（如 ADR-001）→ 状态 → 背景与需求 → 采用的方案 → 备选方案及否定原因 → 决策依据 → 改动范围。

决策状态建议使用：

| 状态 | 说明 |
|------|------|
| 提议中 | 方案还在讨论，不能作为已确定依据 |
| 已采用 | 已确认并按此执行 |
| 已废弃 | 曾经采用或讨论过，但后续不再使用 |
| 被替代 | 被新的决策记录替代，需要注明替代编号 |

### 设计决策沉淀

`10-planning-log.md` 记录「改动前准备怎么做」，`12-design-decisions.md` 记录「项目最终为什么这样设计」。

当某个选择会长期影响产品、交互、架构、数据、API、Prompt 或部署方式时，应该在 `12-design-decisions.md` 中沉淀。尤其是以下情况：

- 放弃了一个看似合理、未来可能再次被提出的方案。
- 当前方案有明确不足，但为了阶段目标、成本或风险接受了这个折中。
- 用户反馈、开发复盘或代码实现证明原决策需要补充、废弃或替代。
- 新成员或 AI 很可能会问「为什么不这么做」。

普通任务进度写入 `05-current-status.md`，每轮修改过程写入 `06-dev-log.md`，改动前方案写入 `10-planning-log.md`，长期设计取舍写入 `12-design-decisions.md`。

## 开发日志格式

每轮开发在 `06-dev-log.md` 中追加一条记录：

```markdown
## YYYY-MM-DD（主题描述）

**触发原因**：这次工作的起因是什么。

**修改内容**：
1. 文件路径 — 改了什么、为什么改
2. ...

**遇到的问题**：
- 问题描述

**解决方式**：
- 如何解决的

**验证方式**：
- 运行了什么测试、命令，或做了什么手动检查

**验证结果**：
- 通过 / 未通过 / 未运行及原因

**本地产物清理**：
- 已删除哪些构建、测试、临时产物；如无产物，写明「无」
```

## Git Commit 格式

```
<type>: <简短描述>
```

| 类型 | 用途 |
|------|------|
| `feat` | 新增功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变外部行为） |
| `docs` | 文档更新 |
| `style` | 样式/格式调整（不影响逻辑） |
| `chore` | 构建、依赖、配置等杂项 |

每次提交对应一个清晰的意图。如果一次改动包含多个独立目的，拆成多个提交。

## 开发规范

以下规则对人、AI 都适用：

- **只改被要求改的地方**。不要顺手"优化"周边代码——未经要求的改动会引入未预期的风险，也让 diff 难以审查。
- **发现需要额外修改时先提出来**，不要夹在同一轮改动中。
- **不要删除注释或文档注释**，除非它们明显错误或有误导性。
- **新增文件或功能模块时**，同步更新对应的 project-log 文档。
- **结束前清理本地产物**。本地构建、测试、预览、导出、截图、缓存、临时脚本等产物，用完后及时删除，避免污染项目目录和系统聚焦搜索。

### 本地构建 / 测试产物清理

每轮开发结束前，必须检查并清理本轮产生的临时文件和可再生成产物，包括但不限于：

- 构建产物：`dist/`、`build/`、`.next/`、`.nuxt/`、`out/`、`coverage/`。
- 测试和缓存产物：`.pytest_cache/`、`.ruff_cache/`、`.mypy_cache/`、`.turbo/`、`.parcel-cache/`、临时数据库、测试报告。
- 调试产物：临时截图、录屏、导出的 PDF/图片、调试日志、一次性脚本、临时 JSON/CSV。
- 预览或验证产物：为了检查效果临时生成、但不属于源码或交付物的文件。

清理规则：

- 源码、模板、正式文档、用户明确要求保留的交付物不要删除。
- 依赖目录是否清理按项目约定执行；例如 `node_modules/`、虚拟环境通常不在每轮结束时删除，除非用户要求或它是本轮临时创建的。
- 如果产物需要保留，必须说明原因，并尽量放入项目约定目录或加入 `.gitignore`。
- `06-dev-log.md` 每轮记录中必须写「本地产物清理」，说明已清理内容或确认无产物。
- AI 不要把临时文件留在项目根目录；需要临时验证时优先使用系统临时目录，结束后删除。

### AI 特别提醒

1. **动手前先理解** — 读懂相关代码和文档后再改，不要基于猜测修改
2. **改动范围克制** — 只做被要求的事，不主动重构、清理、添加"改进"
3. **同步更新文档** — 涉及功能变更、数据库改动、API 新增，同步更新对应文档
4. **遇到矛盾先说出来** — 发现代码或文档有问题，先指出让人确认，不要悄悄改掉
5. **遇到文档不确定时** — 以代码为准，但要告知用户文档与代码的差异，建议更新
6. **收尾要干净** — 本轮生成的构建、测试、调试、预览产物必须清理或说明保留原因，避免影响用户正常使用电脑

### 安全与敏感信息

- 不要把真实 token、password、private key、cookie、连接串密码写入 project-log。
- 示例值必须使用假值，例如 `sk-xxx`、`example-password`、`postgresql://user:password@localhost/dbname`。
- `.env.example` 只记录变量名和占位示例，不记录真实密钥。
- 如果发现敏感信息曾被提交或写入文档，在 `06-dev-log.md` 记录处理方式，并立即轮换对应密钥。

## 主 README 与 project-log 的分工

| 内容 | 主 README | project-log |
|------|-----------|-------------|
| 项目简介 / 功能介绍 | ✅ 写 | ❌ 不写 |
| 快速开始 / 安装指南 | ✅ 写 | ❌ 不写 |
| 技术栈概览 | ✅ 简要列出 | ✅ 详细说明 |
| 数据库设计 | ❌ 不写 | ✅ `02` |
| API 详细定义 | ❌ 不写 | ✅ `03` |
| 架构决策 | ❌ 不写 | ✅ `04` + `10` |
| 产品 / 交互 / Prompt 设计取舍 | ❌ 不写 | ✅ `12` |
| 开发日志 | ❌ 不写 | ✅ `06` |
| 部署详细步骤 | ❌ 不写 | ✅ `07` + `08` |
| 当前进度 | ❌ 不写 | ✅ `05` |
| 代码评审记录 | ❌ 不写 | ✅ `11` |

**原则**：主 README 给「外人」看，project-log 给「自己人」看。
